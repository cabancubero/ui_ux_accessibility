Accessibility Examples for Screen Readers

Examples of properly labeled components for web and mobile apps

Images with Alternative Text (Alt Text)

What is Alt Text?

Alt text is a written description of an image that screen readers announce to users who cannot see the image. It serves multiple purposes: it helps visually impaired users understand image content, appears when images fail to load, and helps search engines understand your content.

Rule of thumb: If you removed the image, would the alt text convey the same information or function? If yes, it's good alt text.

Decorative Image (Empty Alt)

Code: alt=""

When to use: For purely decorative images that don't convey meaningful information (borders, spacers, design elements). An empty alt attribute tells screen readers to skip the image entirely. Never omit the alt attribute completely - always include it, even if empty.

Informative Image (Descriptive Alt)

Code: alt="Task completed successfully"

When to use: For images that convey information or meaning. Describe what the image shows or what information it communicates, not just "image of..." or "picture of...". Focus on the purpose, not unnecessary details.

Alt Text Best Practices:

  • Keep it concise (typically under 125 characters)
  • Don't start with "image of" or "picture of" (screen readers already announce it's an image)
  • For complex images like charts, provide a brief alt text and a longer description nearby
  • For images of text, the alt text should include that exact text
  • Context matters - the same image might need different alt text in different situations

Icon Buttons with ARIA Labels

What are ARIA Labels?

ARIA labels provide invisible text descriptions for interactive elements that screen readers announce to users. They're essential when an element's purpose isn't clear from its visible content alone - like icon-only buttons. Think of them as invisible sticky notes that only assistive technology can read.

Key principle: If a button or link has visible text already, you DON'T need an aria-label. Use aria-label when the visual appearance doesn't communicate the function to someone who can't see it.

Example 1: Notification Button

aria-label="View notifications"

Without this, a screen reader would only say "button" - not helpful! The aria-label tells users what the button does.

Example 2: Notification Badge

aria-label="3 unread notifications"

Dynamic aria-labels can provide real-time context. The badge updates to announce "3 unread notifications" when the count changes.

Example 3: Toggle Button with State

aria-label="Open menu" / "Close menu" aria-expanded="false" / "true"

For toggle buttons, update the aria-label to match the current state. The aria-expanded attribute tells screen readers whether the controlled content is open or closed.

ARIA Label Best Practices:

  • Use action verbs that describe what happens when clicked (View, Open, Delete, Add)
  • Keep labels concise but descriptive enough to be meaningful
  • Update dynamic labels when the button's function changes
  • Don't duplicate visible text - if text is already there, aria-label will override it unnecessarily
  • Combine with title attribute for mouse hover tooltips, but don't rely on title alone
  • Test with actual screen readers - what sounds good visually might be awkward when spoken

Form Inputs with Proper Labels

Why Input Labels Matter

Proper input labels are critical for accessibility. They create a programmatic association between the label text and the input field, so screen readers announce what each field is for. Labels also improve usability for everyone - clicking the label focuses the input, making it easier to use on mobile devices.

Every input MUST have a label. If you can't show a visible label, use aria-label, but visible labels are always preferred because they help all users, not just screen reader users.

Method 1: Explicit Association (Recommended)

<label for="username">Username</label> <input id="username" type="text" />

The for attribute on the label matches the id on the input. This creates a programmatic link that screen readers understand. Users can click the label to focus the input.

Method 2: Labels with Additional Help Text

<label for="email">Email Address</label> <input id="email" aria-describedby="email-help" /> <p id="email-help">Help text here</p>

Use aria-describedby to link help text or error messages. Screen readers will announce the label first, then the description. This provides context without cluttering the main label.

Method 3: aria-label (When No Visible Label)

<input type="search" aria-label="Search products" />

Use aria-label only when design constraints prevent a visible label. Common cases: search boxes where the button provides context, or filters in a data table where column headers serve as labels. Visible labels are always better when possible.

Input Label Best Practices:

  • Always use visible labels when possible - they help everyone, not just screen reader users
  • Place labels above or to the left of inputs for better readability
  • Make labels descriptive - "Email" is better than "Enter your email address here"
  • Use aria-describedby for help text, error messages, or format requirements
  • Don't use placeholder text as the only label - it disappears when typing and has low contrast
  • For required fields, add "required" to the label or use aria-required="true"
  • Group related inputs (like address fields) with fieldset and legend elements

Live Regions and Status Messages

What are Live Regions?

Live regions tell screen readers to announce content changes automatically, even when users aren't focused on that part of the page. This is critical for dynamic updates like notifications, form validation errors, loading states, or success messages. Without live regions, screen reader users would never know these updates happened.

Two politeness levels:

  • aria-live="polite" - Waits for a pause in speech before announcing. Use for non-urgent updates like "Item added to cart" or "Changes saved"
  • aria-live="assertive" - Interrupts immediately. Use only for urgent alerts like "Error: Payment failed" or "Warning: Session expiring"

Live Region Best Practices:

  • Keep messages concise - screen readers will read the entire content
  • Don't overuse assertive - too many interruptions frustrate users
  • Place live regions in your HTML on page load (even if empty initially) so browsers register them
  • Use role="status" for informational updates, role="alert" for warnings/errors
  • Test by turning on a screen reader and triggering the update - does it announce clearly?
  • Consider adding aria-atomic="true" if you want the entire region re-announced on updates
  • Loading spinners should use aria-live="polite" with text like "Loading content"

Comprehensive Best Practices Guide

1. Use Semantic HTML First

Before reaching for ARIA, use the right HTML element. A button element is better than a div with an onclick handler because browsers and assistive technology already know how to handle it. Use nav for navigation, main for primary content, article for self-contained content, and header/footer appropriately.

Remember: "No ARIA is better than bad ARIA." Only add ARIA when semantic HTML isn't enough.

2. Images and Alt Text

Every image must have an alt attribute, even if it's empty. Decorative images get empty alt (alt=""). Informative images need descriptive text that conveys the same information as the image. For complex images like charts or infographics, provide both brief alt text and a longer description elsewhere on the page.

Test: Cover the image and read only the alt text. Does it convey everything important? If yes, it's good.

3. Label Every Interactive Element

Buttons, links, and form inputs must have accessible names. For buttons with text, you're done. For icon buttons, add aria-label. For form inputs, use label elements with for/id associations. If you absolutely can't show a visible label, use aria-label, but visible labels are always preferred.

Test: Tab through your interface with a screen reader. Can you tell what each element does without seeing it?

4. Manage Focus and Keyboard Navigation

All interactive elements must be keyboard accessible. Users should be able to tab through your interface in a logical order. Never use tabindex values greater than 0 (it breaks natural tab order). When content updates dynamically, manage focus appropriately - if a modal opens, move focus into it; when it closes, return focus to the trigger.

Test: Unplug your mouse and navigate your entire application with just the keyboard. Can you reach everything?

5. Announce Dynamic Content Changes

When content updates without a page reload (like notifications, errors, or loading states), screen readers won't automatically announce it. Use aria-live regions to notify users: aria-live="polite" for non-urgent updates, aria-live="assertive" for critical alerts. Combine with role="status" or role="alert" for better semantics.

Test: Submit a form with errors while using a screen reader. Are you notified about the errors?

6. Provide Context for Interactive Widgets

Complex components like tabs, accordions, or custom dropdowns need additional ARIA to work with assistive technology. Use aria-expanded for toggleable content, aria-controls to link controls with their content, aria-selected for tabs, and aria-describedby to provide additional context or instructions.

Test: Can a screen reader user understand the state and relationships between elements?

7. Color and Contrast

Don't rely on color alone to convey information. Add icons, text labels, or patterns. Ensure text has sufficient contrast against backgrounds (4.5:1 for normal text, 3:1 for large text). This helps users with low vision, color blindness, and those using devices in bright sunlight.

8. Test with Real Users and Real Tools

Automated testing catches only about 30% of accessibility issues. Test with actual screen readers like NVDA (Windows, free), JAWS (Windows), VoiceOver (Mac/iOS), or TalkBack (Android). Even better, involve users with disabilities in your testing process.

Quick test: Turn on VoiceOver (Cmd+F5 on Mac) or NVDA and try to complete a key task on your site without looking at the screen.

Remember: Accessibility is not a checklist to complete once. It's an ongoing commitment to making your application usable by everyone, regardless of how they interact with technology.
